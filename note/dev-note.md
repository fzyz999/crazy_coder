#项目目的及基本功能#
由于Dev-Cpp已经基本停止维护，而GUIDE等都无法满足OIer/ACMer/Beginer的使用需求，所以决定开发这个IDE。
这个人群的需求和职业的软件开发者不同。软件开发者开发的往往是一个软件工程，而OIer/ACMer/Beginer等基本上以100～500行的程序为主，程序一般都是在实践算法或数据结构。
所需基本功能大致为：
代码折叠
代码补齐
常见代码段填充
常见资料自动查询
功能强大的调试
对拍器
（跨平台）

#库的选择#
在对速度、开发效率、稳定性等因素进行考虑后，Qt库显然是最佳选择。跨平台库主要有Qt、Wxwidget和GTK。GTK主要为C语言而在windows上不易编译，且不易上手，不利于新人的加入。Wxwidget由于本人没有用过，所以也无从判断。但Qt作为一个商业产品兼开源软件，有着商业软件的一些优点，比如稳定性和易用性等。商业上的成功案例非常多。这是选择Qt的最主要原因。尽管C++语言饱受争议，但其开发效率和上手难度比GTK/GTKmm要低，利于有兴趣共同改进该软件的朋友们的加入。

#版本控制工具#
由于使用github作为托管平台，所以版本控制工具毫无疑问是git。

#文档工具#
Qt自带的qdoc3仅支持C++语言，考虑到未来可能会使用多种语言进行开发，所以选用功能更加强大的Doxygen

#功能及其开发思路#
##语法分析##
语法分析的重点是两个：
首先是执行语法分析的时机。由于处理的是未编写完成的代码，代码会被频繁改动，如果每次改动都重分析一遍效率就太低了。有很多未改动的部分是不必重复分析的。所以，最好是每次只分析用户修改的部分。那么这个怎样实现呢？
用户修改，必然产生键盘输入事件。也就是说，每次codeEditor接到键盘输入事件时，一定是用户进行了修改。这时，对用户输入内容进行语法分析即可。但键盘输入事件每次传入的都是单字符，总不能对单字符进行语法分析吧？没有任何意义啊。那么，设置一个字符串变量记录用户连续几次输入的字符行不行呢？答案是否定的。因为用户前后几次输入可能是在编辑不同的行，所以单纯地把用户的输入连起来是没有任何意义的。我们需要得到的是用户编辑的行的内容，然后重分析这一被改动行。
查阅Qt手册后可以发现QSyntaxHighlighter类的highlightBlock(QString text)函数每次传入的text正是当前行的完整内容，于是，第一个问题就解决了。每次有改动时，被改动行的完整内容会被传入，这样就可以只对被改动行进行语法分析，而不必进行全文语法分析了。
第二是如何实现语法分析
语法分析在很多人眼里很复杂，但实际上并非如此。特别是IDE的editor所需求的不是一个完整的编译器，只是一个能提供自动补齐、语法高亮的简单分析器即可。这个难度就比编译器低多了。例如定义变量的代码：
		int a,b;
基本可以抽象成关键字+变量名
又如for、while循环：
		for(i=0;i<n;i++)
		while(n--)
基本可以抽象成关键字+（条件）等
因此，只要能正确辨别关键字，剩下的工作就很简单了（好像这一段分析基本是废话，如何匹配关键字才是关键，不是吗？）。
那么，如何匹配关键字呢？Qt提供了一个很好的工具：QRegExp类，提供对正则表达式的支持。
以语法高亮的实现为例：
		foreach (const QRegExp& expression, keywords){
			int index = expression.indexIn(text);
			while (index >= 0){
				int length = expression.matchedLength();
				setFormat(index, length, keywordsFormat);
				index = expression.indexIn(text, index + length);
			}
		}
首先，把需要的关键字以正则表达式的形式存好，然后利用QRegExp类提供的indexIn方法，检测待高亮的文本段中有无字符可以匹配上，如果有，则设置一下匹配上的部分的字符粗细/颜色之类的。这样就实现了代码高亮。如果在匹配的同时进行一定的判断，处理，记录等操作，就实现了一个简单的语法分析器。
